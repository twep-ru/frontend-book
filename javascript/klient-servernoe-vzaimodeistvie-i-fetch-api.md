# Клиент-серверное взаимодействие и Fetch API

Почти каждое приложение — от социальной сети до интернет-магазина — получает информацию с сервера и отправляет её обратно. Понимание того, как происходит этот обмен через HTTP-запросы, является критическим навыком для разработчика, так как именно это превращает статичную верстку в живое, динамическое приложение.

### Основы работы с функцией fetch

Для выполнения сетевых запросов в браузере используется встроенная функция `fetch`. Она является мощным инструментом, который работает на основе промисов. Базовый синтаксис функции принимает один обязательный аргумент — URL-адрес ресурса. Поскольку сетевые операции не происходят мгновенно, `fetch` возвращает объект `Promise`, который завершится, когда сервер пришлет ответ.

```javascript
fetch('https://jsonplaceholder.typicode.com/todos/1')
  .then(response => response.json())
  .then(data => console.log(data));
```

Процесс получения данных обычно состоит из двух этапов. Сначала мы получаем объект ответа (`response`), который содержит служебную информацию. Чтобы извлечь полезную нагрузку (сами данные), необходимо вызвать метод `.json()`. Этот метод также возвращает промис, поэтому цепочка `.then()` используется повторно для работы с уже готовым JavaScript-объектом.

### Обработка ответа и статусы

Каждый ответ от сервера сопровождается кодом статуса, который сообщает о результате операции. Коды группы 200 (например, 200 OK или 201 Created) сигнализируют об успехе. Ошибки на стороне клиента (неверный URL или отсутствие прав доступа) относятся к группе 400 (например, знаменитая ошибка 404), а проблемы на стороне сервера — к группе 500.

При работе с `fetch` важно проверять свойство `response.ok`. В отличие от некоторых библиотек, `fetch` не переходит в блок `.catch()` при получении 404 или 500 ошибки — промис считается успешно выполненным, так как запрос физически дошел до сервера и вернулся. Чтобы корректно обработать такие ситуации, разработчику следует вручную проверять успешность ответа и, при необходимости, генерировать исключение.

```javascript
fetch('https://api.example.com/data')
  .then(response => {
    if (!response.ok) {
      throw new Error(`Ошибка: ${response.status}`);
    }
    return response.json();
  })
  .catch(error => console.error('Проблема с запросом:', error));
```

### Методы запросов и передача данных

Хотя метод `GET` используется по умолчанию для получения данных, для создания или изменения ресурсов применяются другие типы запросов. Наиболее распространенным является `POST`. Чтобы отправить данные на сервер, вторым аргументом в `fetch` передается объект настроек. В нем указывается метод, заголовки (`headers`) и тело запроса (`body`).

Заголовки позволяют передать серверу метаданные, например, сообщить, что данные отправляются в формате JSON. Тело запроса обычно формируется путем превращения JavaScript-объекта в строку с помощью `JSON.stringify()`.

```javascript
fetch('https://jsonplaceholder.typicode.com/posts', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json; charset=UTF-8',
  },
  body: JSON.stringify({
    title: 'Новый пост',
    body: 'Текст поста',
    userId: 1,
  }),
})
.then(response => response.json())
.then(json => console.log('Создано:', json));
```

Также данные можно передавать через URL с помощью Get-параметров. Они добавляются после вопросительного знака в формате `ключ=значение`. Если параметров несколько, они разделяются символом амперсанда (`&`). Это стандартный способ фильтрации или поиска данных на сервере.

### Инструменты отладки: вкладка Network

Для анализа сетевой активности в браузерах предусмотрена вкладка Network в инструментах разработчика. Она позволяет видеть все исходящие запросы в реальном времени. В ней можно проверить полный URL запроса, метод (GET, POST и т.д.), полученный код статуса и время выполнения.

Особое внимание стоит уделять подвкладкам:

* Headers: показывает заголовки запроса и ответа.
* Payload: отображает данные, которые вы отправили (актуально для POST-запросов).
* Response: содержит «сырой» ответ от сервера.
* Preview: форматирует ответ (например, JSON) в удобный для чтения древовидный вид.

#### Итог

Изучение Fetch API и принципов клиент-серверного взаимодействия открывает доступ к созданию по-настоящему интерактивных интерфейсов. Мы разобрали, как отправлять базовые запросы, обрабатывать ответы с учетом статус-кодов, передавать данные на сервер в формате JSON и использовать инструменты разработчика для отладки сетевого трафика. Эти знания ложатся в основу работы с любыми современными API и внешними сервисами.

#### Задача на закрепление

Создайте небольшое приложение «Поиск пользователя по ID», используя сервис [JSONPlaceholder](https://jsonplaceholder.typicode.com/).

1. Реализуйте HTML-форму с полем ввода (числовой ID) и кнопкой «Найти».
2. При отправке формы выполняйте `GET` запрос к эндпоинту `/users/{id}`, где `{id}` — значение из поля ввода.
3. Если пользователь найден (статус 200), отобразите его имя (`name`) и email на странице.
4. Если пользователь не найден (статус 404), выведите сообщение: «Пользователь с таким ID не существует».
5. Используйте блок `.catch()` для обработки сетевых проблем (например, отсутствия интернета).

**Критерии успешного выполнения:**

* Запрос отправляется только после нажатия на кнопку (событие `submit`).
* Данные корректно отрисовываются в DOM-дереве.
* Реализована проверка `response.ok` для обработки ошибки 404.
* Код написан чисто, без лишних глобальных переменных.
