# Модули

Разделение кода на более мелкие, логические части — это основа декомпозиции и рефакторинга. Для фронтенд-разработчика понимание модульной системы критически важно, так как это стандартный способ организации кода в современных веб-приложениях, позволяющий делать его читаемым, переиспользуемым и легко поддерживаемым.

### Сущность модулей и изоляция области видимости

Модуль — это обычный файл JavaScript, который отвечает за определенную часть функциональности программы. Цельная программа может состоять из множества модулей, каждый из которых решает свою конкретную задачу. Изначально, если подключить несколько скриптов в HTML обычным способом, они будут делить общую глобальную область видимости. Это означает, что переменная, созданная в одном файле, может быть случайно перезаписана в другом, что неизбежно приводит к трудноуловимым ошибкам.

Чтобы браузер воспринимал файл именно как модуль, при его подключении в HTML необходимо добавить атрибут `type="module"`.

```html
<script type="module" src="script.js"></script>
```

Как только файл объявляется модулем, он получает собственную, изолированную область видимости. Теперь переменные и функции, объявленные внутри него, по умолчанию недоступны извне. Чтобы использовать код одного модуля в другом, применяются специальные механизмы экспорта и импорта.

### Именованный экспорт и импорт

Для передачи сущностей (переменных, функций, классов) наружу используется ключевое слово `export`. Можно экспортировать сущность прямо при ее создании или перечислить все экспортируемые элементы в конце файла.

```javascript
// Переменная доступна для импорта в другие файлы
export const tabsSelectors = {
  tab: '.tab',
  panel: '.panel'
};

const tabsData = [];

// Альтернативный вариант: экспорт в конце файла
export { tabsData };
```

Чтобы получить доступ к этим данным в другом файле, применяется ключевое слово `import`. При именованном импорте названия запрашиваемых сущностей указываются в фигурных скобках и должны точно совпадать с теми, что были экспортированы. Обязательно нужно указывать расширение файла (например, `.js`), если вы работаете в браузере без сборщиков кода.

```javascript
import { tabsSelectors, tabsData } from './tabs.js';

console.log(tabsSelectors.tab);
```

Если в разных модулях сущности называются одинаково, возникает конфликт имен. Решить его можно с помощью переименования на этапе импорта, используя ключевое слово `as`.

```javascript
import { selectors as tabsSelectors } from './tabs.js';
import { selectors as modalsSelectors } from './modals.js';
```

### Экспорт по умолчанию

Каждый модуль может иметь одну главную сущность, которая экспортируется по умолчанию. Для этого добавляется слово `default`. В одном файле может быть только один `export default`.

```javascript
export default function initTabs() {
  console.log('Табы инициализированы');
}
```

При импорте такой сущности фигурные скобки не используются, и вы можете задать ей абсолютно любое имя в файле-приемнике.

```javascript
import initTabs from './tabs.js';
// или можно назвать иначе: import startTabs from './tabs.js';
```

Разработчики часто комбинируют оба подхода: экспортируют главную функцию по умолчанию, а вспомогательные настройки — через именованный экспорт. Однако многие предпочитают использовать исключительно именованные экспорты, так как это снижает путаницу с произвольными именами и улучшает автодополнение в редакторах кода.

### Групповой импорт и реэкспорт

Иногда из модуля нужно забрать сразу множество сущностей. Чтобы не перечислять их все, можно импортировать всё содержимое файла в виде единого объекта с помощью конструкции `import * as`.

```javascript
import * as constants from './constants.js';

console.log(constants.tabsSelectors);
// Доступ к дефолтному экспорту:
constants.default(); 
```

В крупных проектах с глубокой вложенностью папок пути к файлам могут становиться слишком длинными. В таких случаях применяют индексные файлы (`index.js`) и реэкспорт. Если в папке создать `index.js` и написать в нем `export * from './deep-folder/file.js'`, то при импорте из этой папки достаточно будет указать только ее название, и движок автоматически обратится к индексному файлу, значительно сокращая путь.

### Особенности выполнения кода в модулях

Важно понимать несколько неочевидных механизмов работы модулей. Во-первых, сколько бы раз вы ни импортировали один и тот же модуль в разные файлы проекта, код внутри него выполнится ровно один раз при первом обращении.

Во-вторых, хорошей практикой считается не запускать логику (например, вызов функций или мутацию данных) прямо в корне модуля. Модуль должен только объявлять и экспортировать сущности. Вся инициализация должна происходить в единой точке входа — главном файле скрипта.

Кроме того, модули по умолчанию являются отложенными. Браузер загружает их в фоновом режиме и выполняет только после того, как весь HTML-документ будет обработан. Это позволяет безопасно обращаться к элементам DOM-дерева, даже если скрипт-модуль подключен в самом начале страницы в теге `head`.

### Итог

Модули предоставляют мощный инструмент для структурирования приложения. Мы рассмотрели, как изолировать области видимости с помощью атрибута модуля, изучили синтаксис именованного экспорта и экспорта по умолчанию, а также научились переименовывать импортируемые переменные и собирать их в объекты. Понимание того, что модули выполняются единожды и загружаются асинхронно, позволяет строить предсказуемую и надежную архитектуру проекта, грамотно разделяя объявление логики и ее фактический запуск.

### Задача на закрепление

Создайте простую модульную структуру для управления профилем пользователя.

1. Создайте файл `data.js`. Экспортируйте из него по умолчанию объект с полями `username` (строка) и `role` (строка).
2. Создайте файл `permissions.js`. В нем создайте и экспортируйте (через именованный экспорт) функцию `checkAccess`, которая принимает роль пользователя и возвращает булево значение (например, `true`, если роль равна "admin").
3. Создайте главный файл `app.js` (убедитесь, что в HTML он подключен с нужным атрибутом).
4. Импортируйте объект пользователя из `data.js` (задайте ему имя `userProfile` при импорте).
5. Импортируйте функцию `checkAccess` из `permissions.js`.
6. В `app.js` вызовите функцию, передав в нее `userProfile.role`, и выведите результат в консоль.

**Критерии успешного выполнения:**

* Файлы логически разделены и используют правильный синтаксис импорта/экспорта.
* Объект импортируется без фигурных скобок с заданным именем `userProfile`.
* Функция импортируется с использованием фигурных скобок.
* При открытии страницы в браузере в консоль корректно выводится результат проверки доступа.
