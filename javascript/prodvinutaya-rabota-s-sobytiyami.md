# Продвинутая работа с событиями

Глубокое понимание механизмов работы событий позволяет фронтенд-разработчику писать более производительный код, грамотно обрабатывать динамически созданные элементы на странице и выстраивать модульную архитектуру проекта.

### Делегирование событий

Часто возникает задача повесить обработчик клика на множество однотипных элементов, например, на пункты списка задач. Начинающие разработчики могут использовать цикл для обхода всех элементов DOM-коллекции, чтобы добавить слушатель событий к каждому конкретному узлу индивидуально. Однако такой подход имеет два существенных недостатка. Во-первых, если элементов тысячи, браузеру придется хранить в памяти тысячи обработчиков, что негативно скажется на производительности приложения. Во-вторых, современные веб-приложения часто добавляют элементы в DOM-дерево динамически, подгружая их с сервера. Элементы, созданные после того, как ваш код уже отработал и назначил обработчики, не будут реагировать на пользовательские действия.

Решением этих проблем является делегирование событий. Эта концепция заключается в том, что вместо добавления слушателя каждому целевому элементу, мы вешаем один общий обработчик на их общего родителя или на весь документ. Внутри функции обработчика мы определяем изначальную цель клика, обращаясь к свойству `event.target`.

```javascript
document.addEventListener('click', (event) => {
  // Ищем ближайший родительский элемент с нужным классом, включая сам целевой элемент
  const todoItemElement = event.target.closest('.todo-item');
  
  // Если клик произошел внутри нужного элемента, выполняем логику
  if (todoItemElement) {
    todoItemElement.classList.add('is-completed');
  }
});
```

В этом примере использование метода `closest()` имеет критическое значение. Если внутри элемента списка есть вложенные теги (например, текст обернут в тег `span`), клик пользователя может прийтись именно на этот самый глубокий внутренний узел. Метод `closest()` поднимается вверх по DOM-дереву от целевого элемента и находит ближайший узел, соответствующий заданному CSS-селектору. Это гарантирует корректную работу логики вне зависимости от того, на какую внутреннюю часть карточки или пункта нажал пользователь.

### Отмена поведения браузера по умолчанию

Некоторые браузерные события автоматически инициируют заранее определенные платформой действия. Например, клик по ссылке приводит к переходу на другой URL-адрес, а нажатие кнопки отправки (с типом submit) внутри формы вызывает автоматическую перезагрузку страницы с отправкой введенных данных. Во многих случаях, особенно при разработке современных клиентских приложений или при необходимости кастомной валидации полей перед запросом к серверу, это стандартное поведение требуется предотвратить.

Для решения этой задачи используется встроенный метод `preventDefault()`, который необходимо вызвать у объекта события.

```javascript
const form = document.querySelector('.my-form');

form.addEventListener('submit', (event) => {
  event.preventDefault(); // Отменяем автоматическую перезагрузку страницы
  
  // Здесь размещается кастомная логика валидации и отправки данных на сервер
  console.log('Поведение по умолчанию отменено. Форма обрабатывается JavaScript-ом');
});
```

### Генерация собственных событий

Создание кастомных событий — это выразительный инструмент для построения модульной архитектуры фронтенд-проекта. Представьте сложную ситуацию: при загрузке страницы отображается прелоадер, а после его исчезновения должна запуститься серия сложных анимаций контента (появление секций, текста и графики). Если жестко связывать скрипт прелоадера с логикой запуска анимаций контента, это приведет к высокой связанности кода и усложнит поддержку, особенно когда функционал распределен по десяткам разных файлов.

Более элегантный и правильный подход заключается в том, чтобы заставить прелоадер просто сгенерировать уведомление о завершении своей работы. Остальные модули приложения смогут самостоятельно подписаться на это событие и отреагировать на него нужным образом.

```javascript
// Код прелоадера (находится в отдельном модуле)
const preloader = document.querySelector('.preloader');

// Создаем экземпляр кастомного события с помощью класса CustomEvent
// Указываем bubbles: true, чтобы событие могло всплывать вверх по DOM-дереву
// Через свойство detail можно передать любые пользовательские данные в объекте
const preloaderEvent = new CustomEvent('preloaderClose', {
  bubbles: true,
  detail: { animationDuration: 300, isSuccess: true }
});

// Генерируем (диспатчим) событие на конкретном DOM-элементе
preloader.dispatchEvent(preloaderEvent);
```

Для генерации простейших событий можно применять базовый класс `Event`. Однако класс `CustomEvent` является более продвинутым и предпочтительным, так как он предоставляет специальное поле `detail` для отправки полезной нагрузки. Любой другой скрипт на странице теперь может отловить это событие:

```javascript
// Код анимации секций (изолирован в другом файле)
document.addEventListener('preloaderClose', (event) => {
  console.log(`Прелоадер закрыт. Длительность анимации: ${event.detail.animationDuration}мс`);
  
  // Безопасный запуск логики появления контента без зависимости от внутреннего кода прелоадера
});
```

### Итог

Использование паттерна делегирования позволяет кардинально оптимизировать работу приложения и гарантировать надежную обработку динамически внедряемого контента за счет установки единого слушателя на родительский узел. Отмена стандартного поведения браузера через метод `preventDefault()` дает разработчику полный контроль над маршрутизацией и обработкой форм. Генерация собственных событий с использованием `CustomEvent` и `dispatchEvent` способствует созданию независимых, масштабируемых и легко поддерживаемых модулей, которые обмениваются информацией путем диспетчеризации кастомных сигналов сквозь DOM-дерево.

### Задача на закрепление

Создайте интерактивную панель управления задачами, объединяющую делегирование и кастомные события.

1. Подготовьте HTML-разметку: добавьте список `ul` с классом `task-list` и кнопку «Добавить задачу». Внутри списка разместите несколько элементов `li` с классом `task-item`.
2. Напишите логику для кнопки «Добавить задачу»: при каждом клике она должна создавать новый элемент `li` с текстом новой задачи и добавлять его в список.
3. Применяя паттерн делегирования, добавьте один слушатель событий на весь документ, который при клике по любому элементу `task-item` будет переключать на нем класс `completed` (меняющий, например, цвет фона). В процессе обязательно примените метод `closest()`.
4. Реализуйте генерацию кастомного события с именем `taskStatusChanged`. Оно должно запускаться в момент успешного клика по `task-item` и передавать в свойстве `detail` объект с текстом измененной задачи.
5. Зарегистрируйте отдельный слушатель для события `taskStatusChanged` на уровне документа, который будет выводить переданный текст задачи в консоль браузера.

**Критерии успешного выполнения:**

* Динамически созданные элементы корректно реагируют на клики.
* В JavaScript-коде реализован лишь один слушатель события `click` для обработки всех элементов списка.
* При нажатии на задачу в консоли безошибочно отображается ее текстовое содержимое, переданное в слушатель через объект кастомного события.
